[CC解构] 记忆与遗忘：Memory 系统和 Compact 压缩

📅 2026-04-07⏱ 14 min read📝 2629 words

接上回：QueryEngine：驱动一切的主循环。

AI Agent 的一个根本矛盾：上下文窗口是有限的，但工作是无限的。

记忆系统解决跨 session 的持续性——"我上次学到了什么"。Compact 压缩解决单 session 的存活性——"对话太长了，怎么不丢东西地缩短"。

它们是一枚硬币的两面。这篇一起讲。

Part 1: Memory——文件即记忆

四类分类法

Claude Code 的记忆不是自由文本——它有严格的分类：

类型用途示例
user用户偏好和身份"用 bun 不用 npm"、"我是后端工程师"
feedback行为反馈"不要自动 commit"、"解释时用中文"
project项目上下文"这个 repo 是 monorepo"、"deploy 用 Vercel"
reference参考资料"API 文档在 /docs"、"设计规范 link"

每个记忆文件有 frontmatter：

---
name: preferred_package_manager
description: User prefers bun over npm for all JS projects
type: user
---

User explicitly stated on 2026-03-15 that they want bun used 
exclusively. npm commands should be converted to bun equivalents.

这个分类不是装饰——它影响记忆的生命周期和优先级。user 类型的记忆几乎永远有效，project 类型可能随项目变化过期，reference 类型可能需要定期验证。

MEMORY.md：索引，不是内容

MEMORY.md 不是记忆本身——它是索引。每行一条指针，<150 字符：

- [Package Manager Preference](user_pkg_manager.md) — always use bun
- [Testing Strategy](project_testing.md) — vitest + playwright e2e
- [Deploy Process](reference_deploy.md) — Vercel auto-deploy from main

索引有硬性上限：

export const MAX_ENTRYPOINT_LINES = 200
export const MAX_ENTRYPOINT_BYTES = 25_000  // ~125 chars/line

超过 200 行或 25KB，截断并警告：

> WARNING: MEMORY.md is 247 lines (limit: 200). Only part of it 
> was loaded. Keep index entries to one line under ~200 chars; 
> move detail into topic files.

为什么是索引而不是全文？因为 MEMORY.md 会被注入到每一次 API 调用的系统提示词中。如果它是 10KB 的全文，每次调用就多消耗 ~2,500 token。索引模式把开销控制在可预测的范围内。

两步保存流程

保存一条记忆是两步操作：

  1. 写入记忆文件——创建 user_role.md，包含 frontmatter 和详细内容
  2. 更新索引——在 MEMORY.md 添加一行指针

为什么不自动化？因为索引是给模型看的——模型需要知道哪些记忆存在，才能在需要时主动读取。手动维护索引让模型对自己的记忆有"意识"。

什么不该保存

系统提示词明确列出了不该保存的内容：

  • 可从代码推导的信息——代码结构、架构模式、git 历史
  • 临时性信息——当前调试状态、一次性任务上下文
  • 敏感凭证——API key、密码、token

关键思想：记忆存储代码无法表达的东西——偏好、决策背景、人际关系、项目历史。代码能说的，让代码说。

KAIROS：长期 session 的日志模式

当 Claude Code 运行在长期 assistant 模式（KAIROS feature flag）时，记忆策略完全不同：

memory/logs/YYYY/MM/YYYY-MM-DD.md

不再维护 MEMORY.md 索引——改为每天追加日志：

## 2026-04-07

- 15:32 User prefers dark mode in all demos
- 16:45 Discovered auth module uses custom JWT impl, not jsonwebtoken
- 17:20 User explicitly asked to remember: staging URL is api-staging.example.com

日志是只追加的——不重写、不重组。每晚有一个 /dream skill 蒸馏日志到 MEMORY.md 和主题文件。

为什么用日志而不是索引？长期 session 的记忆产生速率远高于交互式使用。索引模式需要每次保存都更新两个文件，日志模式只需追加一行。

一个精妙的缓存细节：系统提示词中的日期用模式（YYYY-MM-DD）而不是今天的日期。因为系统提示词被 systemPromptSection('memory') 缓存——如果嵌入具体日期，午夜跨越时缓存会失效。模型从 currentDate 上下文变量获取实际日期。

Team Memory

const teamMemPaths = feature('TEAMMEM')
  ? require('./teamMemPaths.js')
  : null

Team Memory 在 TEAMMEM feature flag 后面——外部构建不可见。

团队记忆存储在 memory/team/ 子目录下，跟个人记忆分开。个人记忆是私有的（"我喜欢 bun"），团队记忆是共享的（"这个项目用 monorepo"）。

团队记忆要求 auto memory 同时启用——逻辑上合理，因为没有个人记忆系统就没有存储基础设施。

搜索过去的上下文

当记忆文件搜索不够时，Claude Code 还可以搜索 session transcript——.jsonl 格式的完整对话日志：

grep -rn "<search term>" ~/.claude/projects/<slug>/ --include="*.jsonl"

提示词建议用窄搜索词（错误信息、文件路径、函数名），不要用宽泛关键词——因为 transcript 文件很大，搜索很慢。

Part 2: Compact——结构化遗忘

压缩 Prompt：9 段摘要模板

Claude Code 的压缩不是简单的"请总结这段对话"——而是一个精心设计的 9 段结构化模板：

段落内容为什么重要
1. Primary Request用户的显式请求和意图不能丢失目标
2. Key Concepts技术概念和框架上下文基础
3. Files & Code文件路径 + 完整代码片段不能只存摘要
4. Errors & Fixes错误和修复过程防止重复犯错
5. Problem Solving解决问题的思路方法论传承
6. All User Messages所有用户消息（非工具结果）保留用户意图
7. Pending Tasks待完成的任务不能忘事
8. Current Work当前正在做的事连续性
9. Optional Next Step下一步计划方向感

第 6 段是最关键的设计决策——所有用户消息都要保留。因为用户的措辞、语气、反馈是理解意图的最直接证据。工具结果可以丢（可以重新读文件），用户的话不能丢。

Analysis 草稿区

<analysis>
[模型先整理思路]
</analysis>

<summary>
[最终摘要]
</summary>

压缩 prompt 要求模型先写 <analysis>（分析草稿），再写 <summary>（正式摘要）。formatCompactSummary() 在处理结果时会剥离 <analysis> 部分——它只是让模型"想清楚再写"的脚手架。

Chain-of-thought 的又一次应用：先分析再总结，摘要质量更高。但草稿不需要进入上下文——那会浪费 token。

三种方向

方向压缩范围保留范围用途
Full整个对话无上下文快溢出了
Partial (from)最近消息早期上下文最近的工作已不需要细节
Partial (up_to)早期消息最近上下文保留最近的工作细节

Partial Compact (up_to) 的摘要有一个特殊的第 9 段："Context for Continuing Work"——因为摘要后面还有保留的最近消息，读摘要的人需要知道"看完摘要后继续读后面的消息就能跟上"。

NO_TOOLS_PREAMBLE：不许调工具

压缩调用有一段激进的前缀：

const NO_TOOLS_PREAMBLE = `CRITICAL: Respond with TEXT ONLY. Do NOT call any tools.

- Do NOT use Read, Bash, Grep, Glob, Edit, Write, or ANY other tool.
- Tool calls will be REJECTED and will waste your only turn — you will fail.
- Your entire response must be plain text.`

为什么这么强硬？因为压缩 fork 继承了父进程的工具集（为了 cache 共享），但它不应该使用工具。如果模型在压缩过程中调了 cat 去读文件，这个副作用会很难处理。

注释里的数据：Sonnet 4.6 上 adaptive thinking 模型有 2.79% 的概率尝试调工具（vs 4.5 的 0.01%）。把 NO_TOOLS 放在最前面而不是最后面，把出错率压了下去。

尾部还有一段提醒：

const NO_TOOLS_TRAILER =
  'REMINDER: Do NOT call any tools. Tool calls will be rejected.'

头尾夹击。

压缩后的恢复消息

压缩完成后，模型看到的是一段精心措辞的恢复消息：

This session is being continued from a previous conversation 
that ran out of context. The summary below covers the earlier 
portion of the conversation.

如果 suppressFollowUpQuestions 为 true（自动压缩场景）：

Continue the conversation from where it left off without asking 
the user any further questions. Resume directly — do not acknowledge 
the summary, do not recap what was happening, do not preface with 
"I'll continue" or similar. Pick up the last task as if the break 
never happened.

"像中断从未发生过一样继续"——这是用户体验的关键。好的压缩对用户应该是无感的。

如果是 KAIROS 自主模式：

You are running in autonomous/proactive mode. This is NOT a first 
wake-up — you were already working autonomously before compaction. 
Continue your work loop.

Transcript 回溯

压缩后，如果模型需要压缩前的具体细节（精确的代码片段、错误信息），可以读取完整的 transcript 文件：

If you need specific details from before compaction, read the 
full transcript at: [path]

这是一个优雅的设计：摘要提供概览，transcript 提供细节。模型大多数时候只需要概览，偶尔需要回溯细节时再读 transcript。两级存储：热数据在上下文，冷数据在磁盘。

设计原则提炼

1. 索引和内容分离

MEMORY.md 是索引，记忆文件是内容。索引永远在上下文中（200 行上限），内容按需读取。这跟数据库的索引/数据分离是同一个思想。

2. 分类胜过自由文本

四类分类法（user/feedback/project/reference）不是限制——是帮助。没有分类的记忆会变成一锅粥；有分类的记忆可以按类型决定优先级、过期策略、共享范围。

3. 压缩是有结构的遗忘

9 段模板不是在"缩短对话"——是在决定"忘掉什么、记住什么"。所有用户消息必须保留（第 6 段）是一个价值判断：用户的话比 AI 的话重要。

4. 两级存储

热数据（摘要）在上下文窗口，冷数据（transcript）在磁盘。需要时再回溯。这跟 CPU 缓存的 L1/L2 是同一个设计模式。
