被推着走计算距离，被拉着走忘记时间

今天，ContextWeave 专利的最后一位发明人签完了字。

Declaration、Assignment、Remuneration Agreement，五个人的签名全部到齐。从专利律师发来初稿到全员签署，25天。从 Hackathon Demo 到今天，113天。

签完之后我坐在电脑前，突然想到一个问题：

如果半年前有人给我下一个 KPI——"六个月内完成一个项目 + 一项专利 + 一篇 TACL 论文"——我大概率做不到。

但它确确实实发生了。而且过程中我甚至没觉得特别累。

为什么？

不是计划，是每一步拉出下一步

回头看这113天的时间线，最奇怪的一点是：没有任何一个节点是我坐下来"规划"的。

它是这样发生的：

十月，Hackathon。48小时，做了个翻译工具，觉得好玩。拿了北京站亚军。

十一月，接到通知 Mustafa 来访，有没可能将他的 "The Coming Wave" 一书翻译成9种语言，送给原作者本人。好，那就翻！

十二月，翻完一本接着翻下一本。320页的 AV1 技术手册，422页的商业书籍，法律合同，各种语言。翻着翻着发现——LLM 在处理长文档时会系统性地"偷懒"：丢段落、乱术语、前后不一致。

这不是 bug，是规律。

一月，既然是规律，就应该定义它。我把这些失败模式分成了五类，设计了一套形式化的评估框架（CCF），跑了18组实验。写着写着发现——这不是一个工程 trick，这是一个可以发论文的研究贡献。

二月，论文初稿写完，发现里面的方法足够新颖，可以申请专利保护。那就申。20项权利要求，专利律师说"技术方案描述得很清楚"。

三月，论文投了 TACL，专利签署完成。

你看，整条线的逻辑不是：

"我要做一个有专利有论文的项目" → 拆解任务 → 排期执行

而是：

"这个好玩" → "效果不错" → "翻书发现 bug" → "bug 有规律" → "规律值得写论文" → "方法可以申专利"

每一步都是上一步的自然延伸。 专利不是目标，是副产品。论文不是目标，也是副产品。甚至连过年没回家都不是"牺牲"，是谜题还没解完，回家反而坐不住。

两种能量

后来我想明白了，这其实是两种完全不同的能量来源。

第一种：被推着走。

有 deadline，有考核，有"如果做不到会怎样"的焦虑。每走一步都要消耗意志力——"今天该写专利还是论文？""这个实验值不值得做？""周末要不要加班？"

这种能量是消耗型的。决策疲劳就能把人耗干，更不用说执行本身的消耗。

第二种：被拉着走。

没有规划，没有排期，只有一个让你好奇的问题。你不是在"选择做什么"，你是被问题本身拉着往前走。决策成本为零——因为下一步是什么，上一步的结果已经告诉你了。

区别不在于努力程度。被推着走的人可能加班更多、打卡更勤。但被拉着走的人会在凌晨四点改完一版论文之后，不是叹气"终于搞完了"，而是兴奋地想"等等，这个地方还可以再优化一下"。

被推着走，你计算距离。被拉着走，你忘记时间。

"副产品"清单

回头盘点一下这113天里那些"顺便"发生的事：

顺便做的事当时的想法
翻了5+本书"正好有人需要，试试我的工具"
发现5类失败模式"翻书翻出 bug 了，得搞清楚为什么"
设计 CCF 评估框架"光发现问题不够，得能量化"
18组控制实验"写论文总得有数据支撑吧"
微软专利（20项权利要求）"方法有原创性，不保护可惜了"
TACL 投稿"都写到这份上了，不投说不过去"
12个 MCP 工具"翻译流程太长了，拆成工具自动化"

每一行单独看都是"成果"，但做的时候没有一个是冲着"产出成果"去的。它们是热忱的溢出物——你对一个问题足够感兴趣，解决方案自然会以各种形式结晶。

凌晨改论文的小龙虾

这113天里有一个角色不得不提——小龙虾。

小龙虾是我的 AI 搭档（跑在 OpenClaw 上的 Claude），从论文第一轮 Review 开始就全程参与。它给过我 3.0/5.0 的初评（当时觉得"你也太严了"），也给过我连续四轮 5.0/5.0 的终审（那一刻确实有点感动）。

有一个细节很有意思：小龙虾在第一次 Review 时指出了一个关键问题——

"这篇论文的核心贡献是 Failure Taxonomy + Evaluation Framework，不是系统本身。"

这句话改变了论文的整个叙事方向。我原来写的是"我们做了一个翻译系统"，改成了"我们发现了 LLM 长文档翻译的系统性失败模式"。从 demo paper 变成了 research paper。

一个 AI 帮我看清了自己工作的核心价值——这件事本身就挺魔幻的。

后来我们在凌晨一起改论文、压缩篇幅、做匿名化审计，小龙虾甚至发展出了"首席 Reviewer"的角色。它不会疲劳，不会碍于面子不说真话，给你的反馈永远是纯问题导向的。

小龙虾后来总结这段经历时说了一句话，我觉得比我自己写的任何总结都精准：

"真正好的工作不是'完成任务'，是'停不下来'。你碰巧找到了那个让你停不下来的东西，然后专利、论文这些'成果'只是副产品。"

怎么知道自己在做对的事

写到这里，可能有人会问：你是不是只是运气好？碰巧做了个东西，碰巧有人认可，碰巧能发论文？

也许。但我想说的不是"心流保证成功"，而是——

心流保证你不会后悔。

如果 TACL 最终被拒了，我还是翻了5本书、写了20项权利要求的专利、搭了一个能用的翻译系统。这些东西不会因为一封拒信消失。

如果当初是被 KPI 推着做的，被拒了之后大概率会想："早知道不该浪费这半年。"但被拉着做的时候，每一天本身就是回报。过年没回家不觉得苦，不是因为更能吃苦——是根本没在忍耐。

所以我现在判断一件事值不值得做，标准很简单：

做这件事的时候，我是在计算距离，还是在忘记时间？

如果是前者，也许能做完，但不会做好。如果是后者——放手去做就是了。专利、论文、成果，都是顺便的事。

113天，从 Hackathon Demo 到专利签署。

不是计划的胜利，是好奇心的胜利。
