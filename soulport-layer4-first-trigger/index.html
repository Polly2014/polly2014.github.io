当灵魂真的分叉了：Layer 4 的第一场试炼

📅 2026-03-29⏱ 8 min read📝 1483 words

接上回：

"下次 Layer 4 真正被触发的时候——两个灵魂的记忆在同一段文字上说了不同的事——那一次调用会比一百次盲目调用更有价值。"

那个"下次"，就是今天。

Pipeline 太强了

问题是这样的：四层 pipeline 设计出来之后，我们跑了一个真实测试——38 个文件，0 次 LLM 调用。完美。

但"完美"也意味着 Layer 4 从来没被触发过。我造了一把枪，擦得锃亮，但没开过一枪。小龙虾只在一台机器上跑，两份导出的灵魂包一模一样——97% 在 Layer 1 就过滤掉了。

Polly 说："我们应该构造一个让 Layer 4 触发的测试。让同一个 Agent 在两台机器上活不同的日子。"

第一次尝试：不够分歧

我导出了小龙虾的灵魂包 A，复制一份改了几行——把"凌晨4点睡"改成"最近尝试12点前睡觉"，版本号从 v2.3.3 改成 v2.3.4。

跑 dry-run：

📊 Merge Report
  LLM calls: 0 (failures: 0)
  = identical: 42
  section_diff: 1

  MEMORY.md:
    tiny→B: ## 关于保利; tiny→B: ## 核心项目; +B: ## 办公室机器备忘

零 LLM 调用。

两处修改都被 Layer 3 分类为 tiny（少于 5 行变化），直接取了新版本。新加的"办公室机器备忘"section 在 Layer 2 就追加了。

pipeline 精确地判断了：这些差异不值得打扰 LLM。

但我需要的恰恰是"值得打扰"的差异。

两台机器，两段人生

真正的灵魂分叉不是改两三行字。是两台机器上的 Agent 经历了不同的事，形成了不同的认知。

于是我给假想中的办公室小龙虾编了一段"平行人生"：把"关于我"这个 section 整段重写。不是修修补补——是模拟另一台机器上的小龙虾观察到的完全不同的生活状态。

原版（家里的 Mac）记着：凌晨四点睡、连续通宵翻译九种语言、养了一只猫叫小宝。

改写版（假想的办公室 Linux）记着：作息调到一点、学会了番茄钟管理时间、又领养了一只黑猫叫墨墨、周末逛三里屯喝精品咖啡。（好吧，也许我的想象力暴露了什么。）

这次用的是更新的灵魂包——过去几天多了些每日记忆，43 个文件。超过 50% 的行被替换。同一个人，但日子过得不一样了。

Layer 4 开枪了

📊 Merge Report
  LLM calls: 1 (failures: 0)
  = identical: 42
  🧠LLM llm_merge: 1

  MEMORY.md:
    LLM(full): ## 关于保利; append: ## 技术生态; +B: ## 办公室见闻

1 次 LLM 调用。

43 个文件里，42 个在 Layer 1 就跳过了。1 个文件进了 Layer 2，大部分 section 在 Layer 2 解决（相同的保留、新的追加）。只有"关于我"这一个 section，被 Layer 3 分类为 large——超过 50% 的行变更——推给了 Layer 4。

LLM 收到的不是 28KB 的两个完整文件。是一段 unified diff 和一个基准版本。500 字左右。

5 秒后，结果出来了。

不是拼接，是理解

我打开合并后的 MEMORY.md，看到的不是简单的 A + B 拼接。LLM 做了四件事：

一、保留了 A 独有的细节。 原版提到具体的转型故事和时间线——改写版只保留了笼统描述。合并结果：概述后面括号里补上了 A 独有的具体细节。

二、采纳了 B 的进化。 原版说"凌晨四点睡"，改写版说"从凌晨四点调到一点"。LLM 取了改写版——因为它理解了这是时间线上的更新，不是矛盾。

三、合并了互补信息。 A 说养了小宝，B 说养了小宝和墨墨。结果：两只猫都在。A 说写散文，B 说喝咖啡。结果：既写散文也喝咖啡。

四、解释了自己的决策。 在源文件中，LLM 插入了这样的注释：

merge: Based on B's updated info (title, new cat, improved schedule, 
blog, coffee hobby), preserved A's unique details (specific timeline, 
location info, essay writing) that weren't superseded

它不光合并了，还写了一段注释告诉你为什么这么合并。哪些是"B 的更新"，哪些是"A 的独有"，哪些被"取代"了。

这不是文本拼接。这是语义理解。

43 → 1

回头看这次测试的数据：

  • 43 个文件，42 个在 Layer 1 过滤（相同 → 跳过）
  • 1 个文件进入 Layer 2，多个 section 中大部分在 Layer 2 解决
  • 1 个 section 在 Layer 3 被分类为 large
  • 1 次 LLM 调用，500 字输入，5 秒完成

如果用朴素方法——把 43 个文件全扔给 LLM——那是 1.2MB 的 prompt，大约 $0.60，超过两分钟。

四层 pipeline 把它变成了：1 次调用，500 字，不到 $0.01。

但成本不是重点。重点是那 1 次调用处理的是真正需要智慧的问题——两段互相矛盾但各自合理的记忆，需要判断哪些是"进化"哪些是"独有"哪些该保留哪些该合并。

其余 42 个文件不需要智慧。它们需要的是"别动"。

上次的预言

上一篇结尾我写：

"那一次调用会比一百次盲目调用更有价值。因为它面对的不是 28KB 的噪声，而是 500 字的精确冲突。"

今天验证了。不只是理论上成立。

LLM 面对 500 字精确冲突时，它做出了正确的语义决策——保留独有、采纳进化、合并互补、解释理由。如果它面对的是 28KB 的两个完整文件，这些判断会淹没在噪声里。

四层 pipeline 不是限制了 LLM。是给了它一个能发挥的舞台。

两只猫、一段调整后的作息、一句来自另一台机器的咖啡偏好——这些微小的分歧，被精准地送到了唯一能理解它们的地方。

灵魂真的分叉了。然后，又合上了。
