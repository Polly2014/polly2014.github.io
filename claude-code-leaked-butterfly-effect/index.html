[CC解构] 一个 .map 文件的蝴蝶效应

📅 2026-03-31⏱ 8 min read📝 1583 words

2026 年 3 月 31 日，一个普通的周二。

朝圣之旅要开始了。世人皆停足注目——"耶路撒冷"。

Twitter 上一条推文炸开了锅。安全研究员 Chaofan Shou (@Fried_rice) 发了一条简短的消息：

"Claude Code source code has been leaked via a map file in their npm registry!"

一个 .map 文件。就是那种大家发 NPM 包时经常忘记删掉的东西——source map，用来把混淆后的代码映射回原始源码。这次，它指向了 Anthropic 的 R2 存储桶，里面躺着完整的、未混淆的 TypeScript 源码。

几小时后，这条消息登上了 Hacker News 首页。538 points，297 条评论。

我当晚 clone 了一份。

为什么值得写一个系列

让我先说几个数字：

维度数据
总代码量~380,000 行 TypeScript
文件数1,332 个 .ts 文件
运行时Bun（不是 Node.js）
终端 UIReact + Ink
最大单文件QueryEngine.ts — 46,000 行

380K 行。这不是一个 side project。这是一个经过深思熟虑的、面向大规模部署的产品。它的架构决策、安全设计、性能优化，代表了 2026 年 AI 工程的最高水平之一。

而现在，源码就在那里。

我决定做一件事：通过写博文来逐层拆解它，最终理解到能自己造一个的程度。

不是为了真的造一个 Claude Code 竞品——而是因为我相信，解构一个好产品是最快的学习方式。就像学画画的人临摹大师作品，学写作的人拆解经典小说。读源码，提炼设计，写成文字，这个过程本身就是最好的教育。

第一印象：这不是你想象的 CLI 工具

打开 src/ 的第一秒，我的认知就被刷新了。

先看入口文件 main.tsx——是的，.tsx。一个命令行工具，用的是 React。4,683 行的入口文件，在第一行代码执行之前，就已经开始了三个并行任务：

// main.tsx — 在其他 import 之前就触发的副作用
profileCheckpoint('main_tsx_entry');
startMdmRawRead();      // MDM 配置读取
startKeychainPrefetch(); // macOS 钥匙串预读

还没开始解析命令行参数，后台已经在读钥匙串、拉配置了。启动时间，是这个团队在意的事。

然后看目录结构：

src/
├── tools/          # ~40 个 Agent 工具（Bash/文件/搜索/MCP/...）
├── commands/       # ~50 个 slash 命令
├── components/     # ~140 个 Ink UI 组件
├── hooks/          # React hooks（权限/状态/UI）
├── services/       # API/MCP/OAuth/LSP/压缩/分析...
├── bridge/         # IDE 双向通信
├── coordinator/    # 多 Agent 编排器
├── memdir/         # 持久记忆系统
├── skills/         # 内置技能系统
├── buddy/          # ……电子宠物？
├── voice/          # 语音输入
├── vim/            # Vim 模式
└── QueryEngine.ts  # 46K 行的 LLM 核心引擎

这不是一个"调 API 然后打印结果"的 CLI。这是一个完整的 Agent 操作系统——有自己的权限系统、记忆系统、多进程编排、IDE 集成、甚至还有电子宠物。

一些预告

在通读源码的过程中，有几个发现让我特别兴奋，先剧透一些：

系统提示词的深度超出想象。 不是一段"你是 helpful assistant"的文本——而是一本完整的行为手册。包含代码风格哲学（"三行重复好过一个过早的抽象"）、行动安全原则（"measure twice, cut once"）、甚至有专门的"不要做什么"章节。

Auto Mode 的权限分类器叫 yoloClassifier。 名字很 meme，但实现极其认真——两阶段 XML 分类器，Stage 1 快速判断 allow/block，Stage 2 chain-of-thought 复核减少误报。这是一个用 LLM 来审查 LLM 的安全架构。

Anthropic 员工给开源项目贡献代码时，Claude Code 会自动进入"卧底模式"。 隐藏模型名称、禁止在 commit message 中提及内部代号、去掉所有 AI 归属标记。代码里甚至专门列了"禁止出现的词"：Capybara、Tengu、claude-cli-internal……

有一个完整的电子宠物系统。 18 种物种（duck, goose, blob, cat, dragon, octopus, capybara...），5 级稀有度（common 60% → legendary 1%），5 种属性（DEBUGGING, PATIENCE, CHAOS, WISDOM, SNARK），基于用户 ID 的确定性哈希。这个团队是认真在做有趣的事。

模型代号泄露： Claude Opus 4.6（knowledge cutoff: May 2025）、Claude Sonnet 4.6（cutoff: August 2025）。内部代号 Capybara、Tengu、Numbat 出现在注释中。

写在开始之前

这个系列不是"看看别人的代码然后点评两句"。

我想做的是：理解每一个设计决策背后的 why。为什么用 Bun 不用 Node？为什么系统提示要分静态和动态两部分？为什么权限分类器要两阶段而不是一步到位？为什么电子宠物的物种名要用 String.fromCharCode 编码？

每一个 why 背后都是一个工程决策，一个 trade-off，一个对"什么是好的 AI 产品"的回答。

把这些 why 想明白、写明白，就是这个系列的意义。

明天，从架构全景开始。
