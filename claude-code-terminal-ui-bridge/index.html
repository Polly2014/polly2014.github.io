[CC解构] React 渲染到终端：Ink UI 与 IDE Bridge

📅 2026-04-09⏱ 6 min read📝 1189 words

接上回：Skill 系统与 MCP 集成。

当我第一次看到 main.tsx 时，我以为文件名写错了。一个 CLI 工具，入口是 .tsx？

没写错。Claude Code 的终端 UI 是用 React 写的。389 个组件，104 个 hooks——这是一个正经的 React 应用，只不过渲染目标不是浏览器 DOM，而是终端 ANSI 字符。

为什么用 React 做终端 UI

传统的终端 UI 库（blessed、inquirer、prompts）是命令式的：stdout.write('Loading...')，手动管理光标位置，手动清屏重绘。

LLM 交互的特点是：

  • 异步——API 响应是流式的，token 逐个到达
  • 多状态并发——工具在执行、模型在输出、权限在等确认，同时发生
  • 频繁更新——进度条、流式文本、工具结果，每秒更新数十次

React 的声明式模型天然匹配这些需求：

function StreamingResponse({ tokens, toolCalls, permissions }) {
  return (
    <Box flexDirection="column">
      <Markdown>{tokens.join('')}</Markdown>
      {toolCalls.map(tc => <ToolCallUI key={tc.id} call={tc} />)}
      {permissions.pending && <PermissionPrompt />}
    </Box>
  )
}

状态变了，UI 自动更新。不用手动管理"清除上一行→重写→恢复光标位置"。

Ink：React 的终端渲染器

Ink 把 React 组件树渲染成终端输出。Claude Code 不只是使用 Ink——它深度定制了：

  • 自定义 Yoga Layout：native-ts/yoga-layout/（2,578 行），TypeScript 重新实现的 Flexbox 布局引擎
  • 自定义 DOM 层：ink/dom.ts，终端"DOM"的操作接口
  • 自定义事件系统：ink/events/，键盘/鼠标事件处理
  • Focus 管理：ink/focus.ts，Tab 键焦点切换
  • Hit Test：ink/hit-test.ts，鼠标点击区域检测

为什么要重新实现 Yoga？原版 Yoga 是 C++ 写的，需要 native binding。用 TypeScript 重写后：

  1. 不需要编译 native 模块——Bun 直接运行
  2. 包体积更小
  3. 更容易调试和定制

部分组件清单：

IDE Bridge：核心与 Surface 的解耦

Claude Code 的核心不跟任何 UI 绑定——终端、VS Code、JetBrains 都只是不同的 Surface。

双向通信协议

IDE Extension ←→ Bridge Protocol ←→ Claude Code 核心

bridgeMain.ts（2,999 行）实现了这个协议。核心思想：

IDE → Claude Code：

  • 发送用户输入
  • 同步文件变更通知
  • 传递 IDE 上下文（当前打开的文件、选中的代码）

Claude Code → IDE：

  • 流式 AI 响应
  • 工具执行进度
  • 权限确认请求
  • Diff 预览请求

Bridge 组件

文件职责
bridgeMain.ts核心协议实现 (2,999 行)
bridgeMessaging.ts消息编码/解码
bridgePermissionCallbacks.ts权限确认的 IDE 端 UI
bridgeUI.tsBridge 模式的 UI 适配
inboundMessages.ts处理 IDE 发来的消息
inboundAttachments.ts处理 IDE 发来的附件
codeSessionApi.tsSession 管理 API

为什么不直接嵌入终端

VS Code 可以嵌入一个终端窗口运行 CLI——为什么还需要 Bridge？

  1. 原生 UI：权限确认弹窗、diff 预览、进度条——在 IDE 内用原生控件比终端 ANSI 体验好
  2. IDE 上下文：Bridge 可以获取 IDE 知道但终端不知道的信息（当前打开的文件、光标位置、diagnostics）
  3. 双向同步：Claude Code 改了文件后，IDE 需要立即刷新。通过 Bridge 推送比轮询文件系统快
  4. 多 Surface 共存：理论上可以同时从终端和 IDE 跟同一个 Claude Code 实例交互

设计原则提炼

1. 选择渲染框架要看交互模式

LLM 交互是异步流式的——React 的声明式模型天然匹配。如果 Claude Code 是一个"输入命令→输出结果"的传统 CLI，blessed/inquirer 就够了。技术选型要看交互模式，不是看"CLI 该用什么库"。

2. 核心逻辑与 Surface 解耦

Bridge 协议的存在意味着：换一个 IDE（从 VS Code 到 JetBrains），只需要写一个新的 Bridge 适配器，不用改 Claude Code 的核心逻辑。这是关注点分离在 AI 产品中的应用。

3. 值得重新造轮子的时候

TypeScript 重新实现 Yoga Layout——这是一个"造轮子"的正确案例。原因清晰：消除 native binding 依赖、简化构建、减少包体积。不是"我能做得更好"，是"我需要不同的约束"。
