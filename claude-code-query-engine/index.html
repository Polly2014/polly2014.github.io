[CC解构] QueryEngine：驱动一切的主循环

📅 2026-04-06⏱ 15 min read📝 2968 words

接上回：三种并行模式：Fork、Coordinator、Swarm。

如果说架构是骨架，系统提示词是灵魂，工具是手脚——那 QueryEngine 就是心脏。

query.ts 是一个 1,729 行的 AsyncGenerator 函数。它实现了 Claude Code 的核心循环：接收用户输入 → 调 API → 解析响应 → 执行工具 → 把工具结果送回 API → 再解析 → 再执行……直到模型不再返回 tool_use。

这不是一个简单的 while 循环——它处理了自动压缩、流式工具执行、模型降级、token 预算、停止 hook、上下文溢出恢复……每一个边界条件都是真实生产事故的结晶。

全景：五阶段流水线

每一轮 query 循环分五个阶段：

┌─────────────────────────────────────────────────────────┐
│ Stage 1: 上下文管理                                      │
│   Tool Result Budget → Snip → Microcompact →            │
│   Context Collapse → Auto Compact                       │
├─────────────────────────────────────────────────────────┤
│ Stage 2: API 调用                                        │
│   组装消息 → 调用 Anthropic API → 流式接收               │
├─────────────────────────────────────────────────────────┤
│ Stage 3: 流式解析                                        │
│   逐 token 累积 → 检测 tool_use block → 触发执行         │
├─────────────────────────────────────────────────────────┤
│ Stage 4: 工具执行                                        │
│   并发安全分区 → 并行/串行执行 → 收集结果                 │
├─────────────────────────────────────────────────────────┤
│ Stage 5: 循环决策                                        │
│   有 tool_use → 继续循环 │ 无 tool_use → 返回            │
│   max_output_tokens → 恢复 │ stop_hook → 重试            │
└─────────────────────────────────────────────────────────┘

Stage 1：上下文管理——在调 API 之前

每次调 API 之前，query 循环要做大量的上下文管理工作。这是整个函数中最复杂的部分——因为上下文窗口是有限资源，管不好就溢出。

Tool Result Budget

messagesForQuery = await applyToolResultBudget(
  messagesForQuery,
  toolUseContext.contentReplacementState,
  // ...
)

工具结果可能很大（一个 cat 命令输出 10 万行）。Tool Result Budget 把超大的工具结果替换成磁盘引用——模型看到的是"结果已存储，路径在这里"，而不是完整内容。

某些工具被豁免——比如 FileRead 的 maxResultSizeChars 设为 Infinity（持久化读取结果会创建循环引用）。

Snip Compact

if (feature('HISTORY_SNIP')) {
  const snipResult = snipModule.snipCompactIfNeeded(messagesForQuery)
  messagesForQuery = snipResult.messages
  snipTokensFreed = snipResult.tokensFreed
}

Snip 是一种轻量级的上下文裁剪——不调 LLM，只做规则性的裁剪（比如删除过老的消息、截断超长的工具输出）。比 Auto Compact 快得多，但压缩率低。

Microcompact

Microcompact 是 Anthropic 的一个实验性功能（CACHED_MICROCOMPACT feature flag）。它利用 API 的 cache editing 能力——标记旧的工具结果为可删除，让 API 服务端在 cache 中直接删除它们，无需客户端重传整个消息历史。

Context Collapse

另一个实验性功能（CONTEXT_COLLAPSE flag）。跟 Compact 不同，Context Collapse 是一种读时投影——原始消息不变，但渲染给 API 时用折叠后的摘要替换。折叠记录持久化，跨 turn 复用。

Auto Compact

最后的保障网。当上下文接近窗口极限时，调一次 LLM 把整个对话压缩成结构化摘要：

const { compactionResult, consecutiveFailures } = await deps.autocompact(
  messagesForQuery,
  toolUseContext,
  { systemPrompt, userContext, systemContext, ... },
  querySource,
  tracking,
  snipTokensFreed,
)

压缩结果带有详细的遥测：

logEvent('tengu_auto_compact_succeeded', {
  originalMessageCount: messages.length,
  preCompactTokenCount,
  postCompactTokenCount,
  compactionInputTokens,
  compactionOutputTokens,
  compactionCacheReadTokens,
  compactionCacheCreationTokens,
})

注意顺序：Tool Result Budget → Snip → Microcompact → Context Collapse → Auto Compact。每一层过滤掉一些内容，减少后续层的工作量。如果前面的层已经把上下文压到安全线以下，Auto Compact 就不会触发。

Stage 2：API 调用

上下文准备好后，调 Anthropic API：

for await (const message of deps.callModel({
  messages: prependUserContext(messagesForQuery, userContext),
  systemPrompt: fullSystemPrompt,
  thinkingConfig: toolUseContext.options.thinkingConfig,
  tools: toolUseContext.options.tools,
  signal: toolUseContext.abortController.signal,
  options: {
    model: currentModel,
    fallbackModel,
    querySource,
    maxOutputTokensOverride,
    taskBudget: { total, remaining },
    // ...
  },
})) {

几个关键参数：

signal：AbortController 的信号。用户按 Ctrl+C 时，这个信号会触发，中断 API 请求。

fallbackModel：备用模型。如果主模型 API 返回 5xx 或限流，自动降级到备用模型。降级发生时会清空已累积的部分结果（tombstone 机制）。

taskBudget：API 层面的 token 预算控制。total 是整个任务的预算，remaining 是扣除已消耗后的剩余。压缩后需要重新计算——因为服务端看不到压缩前的消息了。

流式降级

if (streamingFallbackOccured) {
  // Yield tombstones for orphaned messages
  for (const msg of assistantMessages) {
    yield { type: 'tombstone', message: msg }
  }
  assistantMessages.length = 0
  toolResults.length = 0
  toolUseBlocks.length = 0
  
  // Discard pending tool execution results
  if (streamingToolExecutor) {
    streamingToolExecutor.discard()
    streamingToolExecutor = new StreamingToolExecutor(...)
  }
}

当主模型流式响应中途失败并降级到备用模型时——已经累积的部分响应要完全丢弃（发送 tombstone 通知 UI 移除）。已经开始执行的工具也要废弃。然后用备用模型重新开始。

Stage 3：流式解析

API 返回的是 Server-Sent Events 流——逐 token 到达。query 循环逐条处理这些事件：

for await (const message of deps.callModel(...)) {
  // 每个 message 可能是：
  // - assistant text block (流式文本)
  // - assistant tool_use block (工具调用)
  // - thinking block (思考过程)
  // - stream events (进度信号)
  
  if (message.type === 'assistant') {
    assistantMessages.push(message)
    // 检测是否包含 tool_use
    for (const block of message.message.content) {
      if (block.type === 'tool_use') {
        toolUseBlocks.push(block)
        needsFollowUp = true
      }
    }
  }
  
  yield yieldMessage  // 实时推送给 UI
}

关键设计：流式解析和 UI 渲染是同步的。模型输出的每个 token 立即推送给 Surface 层渲染——用户能实时看到 AI 在"思考"和"打字"。

背填（Backfill）

if (tool?.backfillObservableInput) {
  const inputCopy = { ...originalInput }
  tool.backfillObservableInput(inputCopy)
  // 只在 backfill 添加了新字段时才 clone
  // 避免修改原始 input（会破坏 prompt cache）
}

某些工具需要在 tool_use block 被 yield 给 SDK/UI 之前补充额外字段——但不能修改原始 input（会破坏 prompt cache 的字节一致性）。所以做一个 clone，只在真正添加了新字段时才替换。

Stage 4：工具执行——StreamingToolExecutor

这是 query 循环中最精妙的工程之一：工具在模型还在输出时就开始执行。

传统方式 vs 流式执行

传统方式：等模型输出完所有 tool_use → 收集所有工具调用 → 按顺序执行 → 返回结果。

Claude Code 的方式：模型输出第一个 tool_use → 立即开始执行 → 模型继续输出第二个 tool_use → 立即开始执行 → 模型输出完毕时，前面的工具可能已经完成了。

class StreamingToolExecutor {
  // 并发安全的工具可以并行执行
  // 非并发安全的工具必须独占执行
  addTool(block: ToolUseBlock, assistantMessage: AssistantMessage): void {
    const isConcurrencySafe = toolDefinition.isConcurrencySafe(block.input)
    // 如果并发安全，立即启动
    // 如果不安全，排队等前面的工具完成
  }
}

并发分区

工具执行有严格的并发控制：

function partitionToolCalls(toolUseMessages, context) {
  // 把工具调用分成多个批次：
  // 批次 1: [FileRead, FileRead, Grep] — 全部并发安全，并行执行
  // 批次 2: [FileWrite]               — 非并发安全，独占执行
  // 批次 3: [FileRead, Grep]          — 并发安全，并行执行
}

最大并发数默认 10（CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY），可配置。

错误传播

如果一个 Bash 工具出错了，它的兄弟进程需要立即停止。StreamingToolExecutor 用一个 child AbortController 来实现：

// Child of toolUseContext.abortController. Fires when a Bash tool errors
// so sibling subprocesses die immediately instead of running to completion.
// Aborting this does NOT abort the parent — query.ts won't end the turn.
private siblingAbortController: AbortController

注意最后一句：中止兄弟不会中止父循环——query 循环还会继续，把错误结果返回给模型让它做下一步决定。

Stage 5：循环决策

工具执行完后，query 循环要决定下一步：

正常路径

有 tool_use block → 继续循环（把 tool_result 送回 API）
无 tool_use block → 返回（模型结束了这一轮对话）

异常恢复

max_output_tokens 恢复：

const MAX_OUTPUT_TOKENS_RECOVERY_LIMIT = 3

如果模型输出被截断（达到 max_output_tokens 限制），循环不会立即退出——它会自动继续，最多重试 3 次。第一次重试前会把 max_output_tokens 提升到 ESCALATED_MAX_TOKENS。

Reactive Compact：

如果 API 返回"prompt too long"错误——不是直接报错，而是触发 Reactive Compact：紧急压缩上下文，然后重试。这比预防性的 Auto Compact 更激进，但只在真正溢出时才触发。

Stop Hook：

用户可以配置 stop hook——在模型结束回答前执行一个 shell 命令。如果 hook 返回 "retry"，query 循环会重新调用 API——模型不知道自己被重试了。

Turn 计数和最大限制

let state: State = {
  turnCount: 1,
  // ...
}

// maxTurns 限制了单次 query 的最大循环次数
// 防止模型陷入无限 tool-call 循环

AsyncGenerator：为什么不是 Promise

query 函数的返回类型是 AsyncGenerator，不是 Promise。这个选择至关重要：

export async function* query(params: QueryParams): AsyncGenerator<
  StreamEvent | Message | TombstoneMessage | ToolUseSummaryMessage,
  Terminal
> {

流式输出。每个 token、每个工具结果、每个进度事件都通过 yield 实时推送。如果用 Promise，调用方只能在整个循环结束后拿到结果。

可取消。调用方可以通过 generator.return() 提前终止循环——比 AbortController 更优雅，因为 using 语句保证了清理逻辑执行。

背压控制。如果 UI 渲染速度跟不上模型输出速度，yield 会自然暂停——这是 AsyncGenerator 的内置背压机制。

思考规则：巫师的咒语

代码中有一段充满幽默感的注释：

/**
 * The rules of thinking are lengthy and fortuitous. They require plenty 
 * of thinking of most long duration and deep meditation for a wizard to 
 * wrap one's noggin around.
 *
 * The rules follow:
 * 1. A message that contains a thinking block must be part of a query 
 *    whose max_thinking_length > 0
 * 2. A thinking block may not be the last message in a block
 * 3. Thinking blocks must be preserved for the duration of an assistant 
 *    trajectory (a single turn, or if that turn includes a tool_use 
 *    then also its subsequent tool_result and the following assistant)
 *
 * Heed these rules well, young wizard. For they are the rules of thinking, 
 * and the rules of thinking are the rules of the universe. If ye does not 
 * heed these rules, ye will be punished with an entire day of debugging 
 * and hair pulling.
 */

翻译：thinking block 的生命周期管理很复杂，搞错了要调一天 bug。这段注释用幽默包装了一个严肃的工程约束——thinking block 必须跟随特定的消息序列，否则 API 会报错。

Memory Prefetch：隐藏的延迟优化

query 循环一开始就启动了一个内存预取：

using pendingMemoryPrefetch = startRelevantMemoryPrefetch(
  state.messages,
  state.toolUseContext,
)

这个预取跟主循环并行运行——在模型思考和输出的同时，后台在搜索相关的 memory 文件。当工具执行结束、需要准备下一轮 attachment 时，memory 搜索可能已经完成了。

using 关键字保证了资源清理——无论 generator 是正常结束、抛异常、还是被 .return() 取消，cleanup 都会执行。

设计原则提炼

1. 五层上下文管理是防御纵深

不是一种压缩方案——是五种，从轻量到重量，层层过滤。Budget → Snip → Microcompact → Collapse → Auto Compact。大部分情况下前几层就够了，Auto Compact（最贵）很少触发。

2. 流式一切

从 API 响应到工具执行到 UI 渲染——全链路流式。用户看到的延迟是"第一个 token 到达的时间"，不是"整个响应完成的时间"。AsyncGenerator 是实现这一点的核心语言特性。

3. 错误恢复不是可选的

max_output_tokens 恢复、流式降级、Reactive Compact、Stop Hook 重试——query 循环为每种可能的失败场景都有恢复路径。LLM 交互的不确定性意味着：如果你不处理异常，异常就会成为常态。

4. 巫师的智慧：把约束写成文档

thinking block 的规则用幽默的文风写成了注释。好的工程团队知道——最危险的不是代码复杂，而是约束没有被记录。一段幽默的注释比一个安静的 bug 好一万倍。
