[CC解构] Skill 系统与 MCP 集成

📅 2026-04-08⏱ 9 min read📝 1768 words

接上回：记忆与遗忘：Memory 系统和 Compact 压缩。

一个 Agent 的能力如果写死在代码里，它就只能做开发者预想到的事。Claude Code 用两个系统打破这个限制：Skill（内置的可复用能力模板）和 MCP（外部工具的标准接入协议）。

Part 1: Skill——可复用的能力模板

三层来源

Claude Code 的 Skill 有三种来源：

来源位置可见性示例
Bundled编译在二进制里全部用户/部分 Ant-only/verify, /loop, /simplify
Project.claude/skills/当前项目/commit, /deploy
User~/.claude/skills/当前用户自定义工作流

Bundled Skills

看看已知的 bundled skills：

batch.ts        — 批量执行
loop.ts         — 循环调度（→ cron）
remember.ts     — 保存记忆
simplify.ts     — 审视改动
verify.ts       — 验证实现（Ant-only）
stuck.ts        — 卡住时的自助
debug.ts        — 调试辅助
keybindings.ts  — 快捷键配置
claudeApi.ts    — Claude API 使用指导
skillify.ts     — 把操作转化为 skill

每个 bundled skill 是一个 BundledSkillDefinition：

type BundledSkillDefinition = {
  name: string
  description: string
  allowedTools?: string[]     // 限制可用工具
  model?: string              // 指定模型
  context?: 'inline' | 'fork' // 在当前上下文还是 fork
  agent?: string              // 指定 agent 类型
  files?: Record<string, string>  // 参考文件
  getPromptForCommand: (args, context) => Promise<ContentBlockParam[]>
}

关键字段：

allowedTools——限制 skill 能使用的工具。/verify 只能读不能写（防止验证过程修改代码）。

context: 'fork'——在 fork 子进程中执行。skill 的工作不会污染主对话上下文。

files——参考文件。第一次调用时懒加载提取到磁盘，模型可以 Read/Grep。有安全的路径校验（防止 .. 遍历）和 O_NOFOLLOW 防止 symlink 攻击。

/loop：自然语言到 Cron

/loop 是我最喜欢的 bundled skill 之一——它把自然语言变成 cron 调度：

/loop 5m /babysit-prs          → */5 * * * *
/loop check the deploy          → */10 * * * * (默认 10m)
/loop check the deploy every 20m → */20 * * * *
/loop 2h run tests              → 0 */2 * * *

解析规则只有三条优先级：

  1. 首 token 匹配 ^\d+[smhd]$ → 那是间隔，剩余是 prompt
  2. 尾部 "every" 子句 → 提取间隔，剥离 prompt
  3. 默认 10 分钟

然后是间隔到 cron 表达式的映射表——包括处理"不能整除"的情况（7 分钟→取最近的整除值并告诉用户取了整）。

30 行解析规则，覆盖了 95% 的日常用法。这就是好的 prompt engineering——不是写一个通用 NLP 解析器，而是用简单规则覆盖主要场景。

/verify：Ant-only 的深度验证

export function registerVerifySkill(): void {
  if (process.env.USER_TYPE !== 'ant') return  // 仅内部用户
  registerBundledSkill({ name: 'verify', files: SKILL_FILES, ... })
}

/verify 的提示词跟 Verification Agent 密切相关——它调用 verification agent 来对代码做对抗式验证。Ant-only 是因为它依赖内部 MCP 工具（如 claude-in-chrome 浏览器自动化）。

Skill 注册流程

const bundledSkills: Command[] = []

function registerBundledSkill(definition): void {
  bundledSkills.push(command)
}

全局注册表，模块初始化时注册。getBundledSkills() 返回副本防止外部修改。

Part 2: MCP——标准化的外部工具

MCP Client 架构

Claude Code 作为 MCP Client，可以连接多个 MCP Server。每个 Server 提供一组工具，动态注册到 Claude Code 的工具池中。

services/mcp/ 目录有 23 个文件，覆盖了完整的 MCP 生命周期：

文件职责
client.tsMCP 连接管理
config.tsServer 配置解析
normalization.ts工具名标准化
channelPermissions.ts工具权限管理
channelAllowlist.ts工具白名单
auth.tsOAuth 认证流
officialRegistry.ts官方注册表
InProcessTransport.ts进程内通信
MCPConnectionManager.tsxReact 组件化的连接管理

工具名标准化

MCP 工具的命名遵循 mcp__<server>__<tool> 格式：

// normalization.ts
// 避免不同 server 的工具名冲突
// 例如：mcp__github__create_issue
//       mcp__linear__create_issue

有一个特殊模式 CLAUDE_AGENT_SDK_MCP_NO_PREFIX——SDK 模式下不加前缀，因为 SDK 用户自己管理工具名空间。

连接管理

MCPConnectionManager.tsx——是的，连接管理器是一个 React 组件。它用 React 的状态管理来追踪每个 MCP Server 的连接状态（pending/connected/disconnected/error），UI 层可以直接订阅状态变化。

连接支持多种传输方式：

  • stdio——标准输入输出（最常见，本地 MCP Server）
  • HTTP——Streamable HTTP（远程 MCP Server）
  • InProcess——进程内通信（内置 MCP Server 如 VS Code SDK）

权限和安全

MCP 工具跟内置工具共享完全相同的权限框架：

// MCPTool 的 isMcp 标记
isMcp?: boolean
mcpInfo?: { serverName: string; toolName: string }

分类器、权限检查、sandbox 约束——一切都走标准流程。不存在 MCP 工具的特权绕过。

channelAllowlist.ts 实现了工具白名单——企业管理员可以限制哪些 MCP Server 的哪些工具可以使用。

官方注册表

// officialRegistry.ts
// Anthropic 维护的"官方" MCP Server 列表
// 类似于 npm 的 scope 认证

用户可以从官方注册表一键安装 MCP Server，不用手动配置连接参数。

Skill + MCP 的协作

Skill 可以引用 MCP 工具。比如 /verify skill 的提示词会检查是否有 mcp__claude-in-chrome__* 或 mcp__playwright__* 可用——有的话就用浏览器自动化做前端验证。

Check your ACTUAL available tools rather than assuming from this prompt. 
You may have browser automation (mcp__claude-in-chrome__*, mcp__playwright__*), 
WebFetch, or other MCP tools.

这种"检查你有什么工具，然后用它"的模式，让 Skill 能够适应不同的环境配置——有浏览器自动化就用浏览器，没有就用 curl。

设计原则提炼

1. 三层来源 = 三种治理

Bundled skills 由 Anthropic 维护（代码审查、安全检查）。Project skills 由团队维护（项目规范）。User skills 由个人维护（个人工作流）。三层分离意味着三种不同的信任级别和更新节奏。

2. Prompt 就是代码

每个 skill 的核心是一段 prompt——getPromptForCommand() 返回的文本。Skill 系统本质上是一个 prompt 的包管理器——封装、版本化、分发提示词。

3. 外部工具零特权

MCP 工具走完整的权限框架，没有快车道。这是"安全不是事后补丁"原则在插件层面的体现。

4. 适应性优于预设

/verify 不假设你有浏览器自动化——它检查你是否有，然后适配。好的 skill 设计是发现能力然后利用，不是假设能力然后失败。
