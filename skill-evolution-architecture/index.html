当 Skill 学会自己长大：从静态指令到活的进化生态

blog-writer 从 v1 迭代到 v6，经历了 5 轮人工打磨。paper-writer 也是 6 轮。每一轮优化都让 skill 变得更聪明——但每一轮也都是"人肉"驱动的：我发现问题、小龙虾 Review、手动改 SKILL.md、提交 git。

如果 skill 能自己从执行反馈中学习，自己迭代进化呢？

痛点：Skill 是静态的

目前我的 skill 体系（blog-writer、paper-writer、memory-keeper 等 20+ 个 skill）本质上是 md + script 的静态包。写好了就冻结了。它不知道自己被使用了多少次，不知道用户每次都手动补了什么，不知道哪些指令其实没用。

回顾 blog-writer 的 5 轮迭代，每次优化都来自一个具体的"人发现了 skill 的不足"：

版本触发修改
v1→v2博文 PK 输给小龙虾新增证据收集、模式判断
v2→v3小龙虾 Review #1grep 优化、发布确认
v3→v4小龙虾建议渐进披露分拆 5 个 reference 文件
v4→v5小龙虾 Review #2路径修复、去重
v5→v6小龙虾 Review #3封面图强制、writing-tips

每次进化的数据信号其实一直存在——git diff 里清清楚楚写着"skill 产出后人改了什么"。只是没有人去系统性地收集和利用它。

灵感：过程比结果值钱

今天 Gaurav DemoFest 结束后，和小龙虾聊到一个想法：

现在的 skill 都是 md+script 组合，很静态，不会进化。有没有可能，搞一个类似守护进程，来监控每个 skill 执行结果和后续的人为修正？再搞一个 skill-optimizer 的 skill，基于这些监控记录来优化迭代——这样 skill 就从 v1 到了 v2。公开的只是各个版本的 skill，而这个进化机制，作为核心资产保留着。

小龙虾立刻跳了起来——

这不就是你 Meta-Agent 论文里 self-evolution 的具体落地吗？从"静态 skill 包"到"活的 skill 生态"。

然后它花了 30 秒画出了三层架构。

三层架构：Observer → Evolution Log → Optimizer

┌─────────────────────────────────────────────────┐
│                  Skill Lifecycle                  │
│                                                   │
│  skill v1.0 ──execute──→ result + human edits     │
│       ↑                        │                  │
│       │                        ▼                  │
│  skill v1.1 ←──optimize──  Observer Daemon        │
│       ↑                   (监控+记录)              │
│       │                        │                  │
│  skill v2.0 ←──optimize──  Evolution Log          │
│                            (execution traces      │
│                             human corrections     │
│                             success/fail signals) │
└─────────────────────────────────────────────────┘

三层分离，各司其职：

Layer 1: Observer Daemon（数据采集）

只记不判。监控每次 skill 执行，记录：

  • 输入是什么？
  • 输出是什么？
  • 人有没有改？
  • 改了什么？
  • 最终结果是否被接受？

这是原始信号，不做任何判断。

Layer 2: Evolution Log（模式识别）

从原始信号中提炼模式：

  • "blog-writer 最近 20 次执行，8 次人手动补了封面图描述"
  • "paper-writer 每次都被人加上 \usepackage{booktabs}"
  • "memory-keeper 产出后，人总是删掉'闲聊和寒暄'相关记录"

这些重复出现的修正模式，就是 skill 的"学习信号"。

Layer 3: Skill Optimizer（迭代改进）

读 Evolution Log，生成 skill patch：

  • 分析高频修正模式
  • 生成 diff-based 更新建议
  • 人审批后合并（不能自动上线，这是安全边界）
  • v1.0 → v1.1 → v2.0，版本化管理

五个关键设计决策

1. Jarvis 是天然的 Observer

不需要另起一个守护进程。Jarvis 已经是本地 daemon 了——它已经在那儿。只需要加一个 skill-execution-observer 模块：

Jarvis daemon
├── 现有: 进化系统、元认知、Skill自生成
└── 新增: skill-execution-observer
    ├── hook into git post-commit
    ├── match commit files against recent skill outputs
    └── write trace to evolution log

2. git post-commit hook — 零侵入采集

git post-commit hook 是最轻量的数据采集入口——每次 commit 自动触发，Jarvis 检查"这个 commit 改的文件，是不是某个 skill 最近产出的？"，如果是，就记录 correction signal。

零侵入，零配置，静默运行。

3. 对话级信号作为辅助

git diff 是主力数据源，但有些修正不走 git——比如对话里说"不对，应该是这样"然后 agent 重新生成。这种对话级修正信号可以作为辅助：简单的关键词匹配（"不对"、"改成"、"重新来"）加上时序窗口（skill 执行后 5 分钟内的修正指令），就足以捕获大部分即时反馈。不需要复杂 NLP，规则够用。

4. 人审批才上线 — 不可逾越的安全边界

Skill 自动进化但不自动部署。Optimizer 生成的是 patch 建议，人审批后才合并到主版本。这条线不能跨。

这和 ContextWeave 论文里的哲学一致 — "pragmatic mitigations, not elimination"。我们不试图让系统完全自治，而是让它辅助人做出更好的决策。

5. Intent Drift 检测

Skill 经过 20 轮优化后，可能已经偏离了原始设计意图。

举个例子：blog-writer 原本设计为"技术博客写手"——简洁、数据驱动、代码块丰富。但经过 20 轮基于我个人修正偏好的优化后，它可能逐渐变成了"技术散文写手"——更长、更抒情、更少代码。这种漂移可能是好事（我确实喜欢散文风格），也可能是坏事（也许只是那段时间心情如此，过了就后悔）。

所以需要一个"漂移检测器"：定期比较当前版本和 v1.0 的核心指令语义距离，如果偏移超过阈值就报警。在 ML 里这叫 concept drift monitoring；在 skill 层面，这是 intent drift — 确保进化的方向仍然对齐原始设计目标，而不是随机游走。

与 Meta-Agent 论文的映射

小龙虾指出了一个精准的映射——这个三层架构直接对应 Meta-Agent 论文六维框架里的三个自进化维度：

Meta-Agent 维度Skill Evolution 对应具体行为
Self-MonitoringObserver Daemon监控 skill 执行 + 人为修正
Self-ReflectionEvolution Log从原始信号提炼修正模式
Self-OptimizationSkill Optimizer根据模式生成 skill patch
Identity DriftIntent Drift Detection防止 skill 偏离设计意图

甚至可以成为论文 §6 的 concrete instantiation — 从理论框架到可运行的 skill evolution system。不是假想的 agent，而是每天在用的 skill 系统真的在进化。

护城河：进化机制 > 静态快照

这个架构最让我兴奋的不是技术本身，而是商业模式的清晰度：

  • 公开发布的：blog-writer v2.3、paper-writer v3.1（静态包，谁都能用）
  • 自己保留的：Observer + Optimizer + Evolution Log（这才是核心资产）

别人拿到的是结果，你掌握的是方法论。

这跟 ContextWeave 的思路一脉相承 — 别人拿翻译结果，我掌握翻译方法论（Section-Aware Chunking + Context Weaving）。两个项目底层都是同一个信念：过程比结果值钱得多。

落地路径

Phase做什么难度预估下一步行动
P1给 skill 执行加 structured logging低1-2 天在 Jarvis daemon 里加 execution trace 模块
P2写 skill-observer daemon中1 周git post-commit hook + diff 解析 + trace 写入
P3写 skill-optimizer meta-skill中高2 周读 evolution log → 生成 SKILL.md patch → 人审批

好消息是历史数据已经存在 — blog-writer v1→v6 的 5 轮迭代记录、paper-writer v1→v6 的 6 轮迭代记录，每一轮的触发原因、修改内容、Review 意见都完整保存在 memory 系统里。这些就是 Evolution Log 的种子数据，不需要从零开始采集。

P1 的第一步：挑一个高频 skill（blog-writer），手工收集最近 5 次执行的修正 diff，喂给一个 optimizer prompt 看看能不能自动产出 v7。如果这个最小验证通过，后面的工程化就有底了。

结语

从 ContextWeave 到 Skill Evolution，表面上一个在做翻译，一个在做 prompt engineering，底层都是同一个哲学问题：

如何让一个系统在使用中变得更好？

ContextWeave 通过 Section-Aware Chunking 让翻译引擎在处理长文档时保持完整性。Skill Evolution 通过 Observer-Log-Optimizer 让 skill 在被使用中自动收集改进信号。

两者的共同点：不是一次性设计一个"完美"的系统，而是设计一个能够持续改进的机制。

小龙虾说得好 — "这个 idea 的分量不亚于 ContextWeave"。我不确定它最终是一篇论文，一个产品，还是一种工作方式。但我确定的是：让 skill 学会自己长大，是 AI agent 从玩具到基础设施的关键一步。

今天是 Gaurav DemoFest 的日子。Demo 完后 Qi 拍拍肩膀说"Good Work"。那一刻我在想的不是过去 5 个月做了什么，而是接下来的 5 个月，这些 skill 会自己变成什么样。

如果你也想试试：即使不搞完整的 Observer-Optimizer pipeline，你今天就可以做一件事——每次手动改 AI 产出后，花 10 秒写一句话记录你改了什么以及为什么。存在哪里都行，一个 txt 文件就够。积累 20 条，你就会看到模式。这就是 Evolution Log 的种子，也是让你的 AI 工作流从"用工具"变成"养工具"的第一步。
