Polly Chat v3.5：让思考被看见

📅 2026-05-13⏱ 14 min read📝 2699 words

系列回顾：2.0 零后端重构 → Vision 多模态 → D1 持久化 → v2.5 欢迎屏 + IndexedDB → 3.0 自己翻博客 → 本篇 v3.5

Polly Chat 进化树更新一格——v3.5 落地。

一棵慢慢长出来的树

往回数，这个挂在博客首页右下角的小聊天框，已经偷偷长了大半年：

v1.0  (2025-02)  ─  原始形态：FastAPI 后端 + DeepSeek 直连，问一句答一句
v2.0  (2026-02)  ─  零后端：直连 CopilotX，SSE 流式，memory 系统动态生成人设
v2.1  (2026-02)  ─  视觉：支持粘贴/上传图片，多图并发
v2.2  (2026-02)  ─  记忆：D1 持久化 + 回头客识别（"你之前问过 X..."）
v2.5  (2026-02)  ─  欢迎屏 + IndexedDB 客户端持久化
v3.0  (2026-04)  ─  Agent + Tools：搜索/阅读/最近博文，让博客自己翻博客
v3.5  (2026-05)  ─  让思考被看见：流式 trace 可视化（本篇）  ✨

几篇前作记下了关键节点：v2.0 把后端拆掉、D1 给它装上记忆、v2.5 加欢迎屏和 IndexedDB，以及一个月前的 v3.0 让它学会自己翻博客。

v3.0 解决的是"知识天花板"——访客问一篇博文，数字 Polly 不再凭 system prompt 硬编，而是真的去 search_blog / get_article 翻一遍。从工程上讲已经挺现代了：worker + agent loop + 三个可调用的博客 tool。

但 v3.0 上线后有个隐藏问题，只有访客真正问到需要查博文的问题时才会暴露——比如"聊聊你最新的博文《镜子里的我》"。

v3.0 的死结：tool 路径会让人干等

复现路径很简单。访客问一句"聊聊你最新的博文《镜子里的我》"：

浏览器 ──► Worker ──► CopilotX (第1轮)：模型决定调 search_blog
                  ◄── 完整响应（必须等完）
            ├─ 执行 search_blog（命中 L1 缓存也要 ~100ms）
            ├─► CopilotX (第2轮)：可能再调 get_article
            ◄── 完整响应（又等一次）
            ├─ 执行 get_article
            ├─► CopilotX (第3轮)：最终 text 答复
            ◄── 这才开始 stream 给浏览器

实测三组 curl：

测试触发 tool？TTFBTotal
"你好"（无 tool）❌4s5s
第一次问博文（冷）✅12.5s25s
第二次问同一篇（热）✅19s33s

第二次比第一次还慢——根本不是缓存冷启动的问题，是结构性的。

handleProxyWithTools 在跑 tool loop 时，每一轮都 await 拿完整的上一轮响应再发下一轮，全部跑完才返回 Response 给浏览器。这意味着 TTFB 等于"所有 LLM 轮次 + 所有 tool 执行时间"的累加。访客在前 12 秒看不到任何东西，浏览器自带的 15 秒 fetch 超时一到，就触发了那张友好的 emoji 错误页：

🤔 Hmm, thinking too hard...

错误文案再可爱，本质还是一个体验事故。

v3.5 的破题：让用户看见，而不是让用户等待

修复方向其实是个二选一题：

方向利弊
A. 应急药把前端 timeout 从 15s 拉到 45s。能止血，但访客仍然要面对一个 24 秒的空白屏。治标不治本
B. 根治立即返回 Response，agent loop 移到后台跑，用 SSE 推自定义事件让前端实时展示思考过程

两个都做了——A 作为最低保障 5 分钟改完，B 作为真正的 v3.5 用了三个小时。本文讲 B。

关键改造一：Worker 立即返回，agent loop 跑后台

新版 handleProxyWithTools 的核心结构变成了这样：

function handleProxyWithTools(body, env, ctx, backend, origin) {
  const { readable, writable } = new TransformStream();
  const writer = writable.getWriter();

  const agentLoop = async () => {
    for (let round = 0; round < 3; round++) {
      await writeStatus('thinking', { detail: '在脑子里翻一翻...' });
      const resp = await fetchCopilotX(...);
      const { firstBlockType, ... } = await detectFirstBlock(resp.body);

      if (firstBlockType === 'text') {
        // 透传 Anthropic 原生 SSE，访客逐字看到答案
        await pipeToWriter(reader, writer);
        return;
      }
      // tool_use → 推送 status 卡片，执行 tool，进入下一轮
      for (const tc of toolCalls) {
        await writeStatus('tool_call', { tool: tc.name, label: humanizeTool(...) });
      }
      const results = await Promise.all(toolCalls.map(tc => executeTool(tc, env)));
      for (const r of results) await writeStatus('tool_result', { count, ok });
    }
  };

  // 关键：用 ctx.waitUntil 包住后台任务，防止 Response 返回后 isolate 提前回收
  ctx.waitUntil(agentLoop());

  return new Response(readable, {
    status: 200,
    headers: { 'Content-Type': 'text/event-stream', 'X-Accel-Buffering': 'no' }
  });
}

三个关键点：

  1. TransformStream + 立即返回 Response(readable)：浏览器的 fetch 在 < 200ms 内就拿到 headers，不再触发任何超时
  2. ctx.waitUntil：Cloudflare Worker 的钩子，明确告诉 runtime"这个 Promise 完成之前别回收 isolate"。没有它，Response 一返回 worker 就可能被关掉，agent loop 中途死掉
  3. X-Accel-Buffering: no + 每 4s 推送 SSE 注释行：双保险防止中间代理（CDN / 反代）误判 idle 把链接掐了

关键改造二：SSE 多事件类型，让前端能"听懂"思考阶段

Anthropic 的原生 SSE 协议里只有 message_start / content_block_delta / message_stop 这些和"答案文本"相关的事件。要让前端展示"现在在思考 / 正在调 search_blog / 找到 3 条结果"这些状态，得给 SSE 加自定义 event 类型。

借鉴了 SpatialQA agent 的设计——SSE 协议本来就允许 event: xxx 行标记事件类型，前端按类型分发：

event: status
data: {"phase":"thinking","detail":"在脑子里翻一翻..."}

event: status
data: {"phase":"tool_call","tool":"search_blog","label":"翻翻博客里有没有 \"镜子里的我\""}

event: status
data: {"phase":"tool_result","tool":"search_blog","count":1,"ok":true}

event: message_start
data: {"type":"message_start","message":{...}}

event: content_block_delta
data: {"type":"content_block_delta","delta":{"type":"text_delta","text":"这"}}
... （这里开始答案逐字流出）

前端拿到 event: status 就建一张 trace 卡片，拿到 content_block_delta 就往答案区追加文字——两条流共存，互不干扰。

关键改造三：卡片合并 + 配对，而不是事件刷屏

第一版的 trace UI 把每个事件都建成独立的一行：

  • 正在思考...
  • 🔍 搜索博客 "镜子里的我"
  • ✓ 完成
  • 正在思考...
  • 📖 阅读《xxx》
  • ✓ 完成
  • 正在整理回答...

视觉很乱。改成卡片合并模式之后：

  • thinking 事件不立即建卡片，而是作为下一个 tool_call 的 pending placeholder。连续多个 thinking 自动合并
  • tool_call + tool_result 配对合并到同一张卡片：从"⟳ 翻翻博客里有没有"镜子""原地变成"✓ 翻翻博客里有没有"镜子" · 1 条结果"

最后访客看到的是一栈整洁的 mini bubble：

┌─────────────────────────────────────────┐
│  ⟳  翻翻博客里有没有 "镜子里的我"        │
└─────────────────────────────────────────┘
┌─────────────────────────────────────────┐
│  ⟳  详细看看《mirror-self-editing-self》 │
└─────────────────────────────────────────┘

[这周还挺高产的，最近这周写了好几篇——...]   ← 答案紧接着流出

当前正在跑的卡片有呼吸边框——蓝色 box-shadow 1.6s 脉冲，告诉访客"这步还在跑"。完成后边框消失，spinner 变成 ✓。流结束后整栈折叠成顶部一行小灰条：

[ ✓ Polly 想了 19 秒    展开 ]

点击展开能看到完整步骤。

关键改造四：trace 持久化（不只是瞬态特效）

刷新页面之后，访客如果还想看一眼之前 Polly 是怎么想的，trace 不能就这么消失。这件事其实可做可不做——很多家的"思考链"也是看完即丢。但选择做的时候有两个理由：

  1. 用户体验：那行折叠 pill 留在历史里比消失要好，回看的成本几乎为零
  2. 科研副产物：D1 里在攒一个"访客提问 → agent in-the-wild trace"的天然数据集，对 ContextWeave 之外的 agent UX 研究有用

实施很轻：

-- D1 messages 表加一列
ALTER TABLE messages ADD COLUMN trace TEXT;

前端 streamResponse 里收集所有 status 事件 + 总耗时，一起塞到 messages.push({role, content, trace: {events, elapsed_ms}})，随 IndexedDB 自动持久化；同时通过 worker 的 /v1/conversations/log 端点同步到 D1。

刷新后 restoreHistory 检测到 msg.trace.events.length > 0，调用 renderSavedTrace(bubble, trace) 直接重建 finalized + collapsed 状态——跳过 spinner 和呼吸动画（不能假装"正在跑"）。后台 dashboard 也加了一个蓝色 mono 字体 pill 来呈现历史推理链：✓ 3 steps · 19s，点击展开看完整 tool 输入参数。

决策表态：访客视图 ≠ debug 视图

写完功能之后做了一轮"哪些不要做"的决策。看似简单，其实是这一版 UI 比 v2.7 高一个台阶的关键：

不做理由
❌ 头像旁加名字 + 时间戳访客是来跟 Polly 聊天的，不是调试 multi-agent。加上反而破坏"和真人聊天"的感觉
❌ trace 卡片 hover 显示完整 tool input/output访客不需要看 raw JSON，那是 debug 视图，留着只增加视觉干扰
❌ 气泡尾部三角小尾巴现代 chat（iMessage、Claude.ai、ChatGPT）都已放弃这种 2010s 设计语言。保留一个 4px 直角已经足够暗示"这是聊天气泡"
❌ 用户头像用户气泡靠右对齐已经足够区分身份，那个灰色 fontawesome 小人头是冗余
❌ done 状态用绿色跟博客主调色板不搭。统一改成中性灰 + 主蓝 rgb(61, 146, 201)

反过来——后台 dashboard chat_observatory 是 debug 视图，那里就留着 raw input/output、mono 字体、详细参数。同一个数据，两套渲染逻辑。

文案口语化：让 Polly 像 Polly

最后一项是文案——把工程师视角的标签改成第一人称口语：

✖ 工程师视角✔ Polly 第一人称
🔍 搜索博客 "x"翻翻博客里有没有 "x"
📖 阅读《x》详细看看《x》
🗓 查看最近 30 天看看最近 30 天写了啥
正在思考...在脑子里翻一翻... / 让我想想...
Thought 19s · 4 stepsPolly 想了 19 秒

工程师习惯说"调用 search_blog tool"，访客感受到的是一个数字分身在翻自己的记忆。这是同一件事的两种叙述，但后者把"工具"翻译成了"行为"。

真正解决的问题，是感知

实测对比：

指标v3.0v3.5
TTFB（tool 路径）12.5s1.7s
后端总耗时（不变）~25s~23s
前端误判 522 超时✅ 必发❌ 不再发生
用户看到的24s 干等 → 突然出文字立刻看到流动的思考链 → 答案逐字流出

后端总耗时几乎没变——LLM 跑 23 秒是物理上限，跑不掉。但感知耗时从 24 秒降到 ~2 秒。

这不是博客 chat 独有的教训。任何后端要跑 ≥ 5 秒的 agent loop，前端都面临同样的取舍：要么等到底再说话，要么把过程拆成结构化事件流出去。让用户看见，而不是让用户等待——这条原则适用于推荐引擎的"正在为你筛选..."、复杂搜索的"已扫描 1.2k 文档..."、长文档翻译的"正在处理第 3/12 章..."。区别只是：你愿不愿意把"思考过程"也设计成产品的一部分。

下一格是什么？v4.0 大概会是真正的范式跃迁——实时 voice 对话，或者多模态 tool（看着访客贴的截图直接回答），或者 agent 之间互相通信。但那是另一篇博文的事了。

眼下的 v3.5 先把基础打好：刷新博客首页，问 Polly："聊聊你最新的博文"。看着那行绿色的 ✓ Polly 想了 19 秒 出现在历史里——这次，等待是看得见的。
