Polly Chat 2.0: From Clunky Backend to Zero-Backend

📅 2026-02-13⏱ 6 min read📝 1062 words

凌晨 1 点，朋友发消息抱怨："你首页那个聊天机器人，太慢了，等半天才出来几个字。"

我打开 Chrome DevTools 看了眼——确实，第一个字符要 5-15 秒，整个回复完要 20+ 秒。这体验，放在 2026 年简直是考古级别。

是时候重构了。

🔍 问题诊断

老架构长这样：

Blog Frontend → Polly-Blog-Backend (Python/FastAPI) → DeepSeek API (付费)

问题在哪？

  1. 非流式响应 - stream=False，必须等 AI 想完才一次性吐出来
  2. 重复加载配置 - 每次请求都重新读 profile_polly.json
  3. 静态人设 - 那个 JSON 文件还是去年写的，早就和我现在的状态脱节
  4. 付费 API - 每月还要花钱买 DeepSeek token

最要命的是——我现在有 CopilotX 了啊！这个"无限续杯大杀器"已经稳定运行在 api.polly.wang，Claude Sonnet 4 免费调用，还支持 SSE 流式输出。

为什么还要维护一个臃肿的 Python 后端？

💡 架构革命

小龙虾（OpenClaw）给了我一个精彩的洞察：

"profile_polly.json 那种静态人设文件完全是上个时代的产物——你已经有了编年体+纪传体的双螺旋记忆系统，几百条记忆、7 个主题、13 天的编年记录，比那个 JSON 里的死板描述丰富 100 倍。"

一语惊醒梦中人！我的 memory/ 目录才是真正的"我"——包含最新的项目进展、今日的思考、核心信念。为什么不从这里动态生成 system prompt？

新架构：

memory/** → build_prompt.py → polly-prompt.json
                                    ↓
Blog Frontend → Cloudflare Worker (chat.polly.wang) → CopilotX (api.polly.wang) → Claude Sonnet 4 (SSE)
                        ↑ API Key 在这里注入

Zero Backend。纯前端。API Key 藏在边缘。

    identity = load_identity()      # 从 topics/digital-polly 提取核心身份

2. 轻量级聊天组件

// static/js/polly-chat.js
class PollyChat {
    async init() {
        const res = await fetch('/polly-prompt.json');
        const data = await res.json();
        this.systemPrompt = data.system_prompt;
    }
    
    async send() {
        // 请求发往 Cloudflare Worker，无需携带 API Key
        const response = await fetch('https://chat.polly.wang/v1/messages', {
            method: 'POST',
            headers: {
                'Content-Type': 'application/json',
                'Accept': 'text/event-stream'
            },
            body: JSON.stringify({
                model: 'claude-sonnet-4',
                system: this.systemPrompt,
                messages: this.messages,
                stream: true
            })
        });
        // SSE 流式解析...
    }
}

3. Cloudflare Worker 代理

直接把 API Key 放在前端？那我的 CopilotX 就成公共厕所了。

解决方案：Cloudflare Worker 作为代理层，API Key 存在加密环境变量里。

// Cloudflare Worker
export default {
  async fetch(request, env) {
    const ALLOWED_ORIGINS = ['https://polly.wang', 'https://www.polly.wang'];
    const origin = request.headers.get('Origin');
    
    // CORS 白名单校验
    if (origin && !ALLOWED_ORIGINS.includes(origin)) {
      return new Response('Forbidden', { status: 403 });
    }
    
    // 代理到 CopilotX，注入 API Key
    const proxyHeaders = new Headers(request.headers);
    proxyHeaders.set('Authorization', `Bearer ${env.COPILOTX_API_KEY}`);
    
    return fetch('https://api.polly.wang' + new URL(request.url).pathname, {
      method: request.method,
      headers: proxyHeaders,
      body: request.body,
    });
  }
};

部署到 chat.polly.wang，前端只知道这个域名，API Key 永远不出现在客户端代码里。

免费额度：Cloudflare Workers 每天 100k 请求，对个人博客来说约等于无限。

4. 模板精简

老版本 index.html：~500 行（WebSocket + HTTP fallback + 打字机效果 + 错误处理）

新版本：~40 行

<script src="&#123;&#123; get_url(path='js/polly-chat.js') &#125;&#125;"></script>

就这么简单。

5. CopilotX CORS 支持

为了让前端能直接调用 api.polly.wang，给 CopilotX 加上 CORS 中间件：

CORS_ORIGINS = [
    "https://polly.wang",
    "http://127.0.0.1:1111",   # 本地开发
]

app.add_middleware(
    CORSMiddleware,
    allow_origins=CORS_ORIGINS,
    allow_methods=["*"],
    allow_headers=["*"],
)

🎯 收获

  1. CopilotX 是基础设施 - 它不只是 CLI 工具，已经成为整个 AI 工具链的核心 API 层
  2. Memory 即人格 - 静态配置文件是"死的"，但记忆系统是"活的"
  3. Zero Backend 思维 - 能前端做的，就别起后端服务
  4. 边缘代理保安全 - Cloudflare Worker 让敏感凭证永远不出现在客户端
