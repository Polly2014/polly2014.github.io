🔍 CopilotX：一个 400 背后藏着两个 Bug

📅 2026-03-28⏱ 7 min read📝 1247 words

周六晚上十点，我让小龙虾帮忙把几张课表截图整理成 Excel。它搞定了——但我想让 Claude Code 也试试。

Claude Code 读完 6 张图，准备回复时：

API Error: 400 {"type":"error","error":{"type":"upstream_error","message":"Bad Request\n"}}

一个裸的 Bad Request，什么信息都没给。

第一层：看得见的 bug

小龙虾翻了 CopilotX 的代码，很快定位了问题。

CopilotX 有两条路由：/v1/responses（OpenAI Responses API）和 /v1/messages（Anthropic Messages API）。Claude Code 走的是后者。关键区别在于——Responses 路由有完整的 vision 检测：

# routes_responses.py — 已有的正确实现
def has_vision_input(body: dict) -> bool:
    for item in body.get("input", []):
        for part in item.get("content", []):
            if part.get("type") == "input_image":
                return True
    return False

检测到图片后，会给 Copilot 后端加上 copilot-vision-request: true header。没这个 header，后端收到 image_url 类型的 content 就直接拒绝。

但 Anthropic 路由？什么都没做。 原样转发，不带 header，后端一脸懵逼地返回 400。

修复不复杂。在 routes_anthropic.py 加一个检测函数：

def _has_vision_content(body: dict) -> bool:
    for msg in body.get("messages", []):
        content = msg.get("content")
        if not isinstance(content, list):
            continue
        for block in content:
            if isinstance(block, dict) and block.get("type") == "image":
                return True
    return False

然后把 vision 参数传进 client：

vision = _has_vision_content(body)
openai_stream = client.chat_completions_stream(openai_payload, vision=vision)

Commit，push，SSH 到 Azure VM，git pull，重启服务。

测试——单张图片，200 ✅。6 张图片一起发，200 ✅。完美。

告诉用户"修好了，再试试"。

第二层：藏在影子里的 bug

用户再跑 Claude Code，还是 400。

什么情况？我刚测完明明好了啊。

拉服务端日志一看，这次报错信息变了：

{"error":{"message":"The requested model is not supported.","code":"model_not_supported"}}

不是 vision 的问题——是模型名的问题。

原来 Claude Code 每轮对话会发两个请求：主模型用 claude-opus-4.6（stream），小模型用 claude-opus-4.6-fast（做内部辅助调用）。主模型的请求成功了，小模型的请求挂了。

翻 CopilotX 的模型映射逻辑，bug 立刻现形：

def map_anthropic_model_to_copilot(model: str) -> str:
    if model in ANTHROPIC_TO_COPILOT_MODEL_MAP:
        return ANTHROPIC_TO_COPILOT_MODEL_MAP[model]
    # 如果名字里有点号（如 4.6），认为已经是 Copilot 格式，原样返回
    if "." in model:
        return model  # ← 问题在这里

claude-opus-4.6-fast 里有 .（因为 4.6），所以 mapper 以为这已经是合法的 Copilot 模型名，原样返回。但 Copilot 后端压根不认 claude-opus-4.6-fast 这个名字。

"有点号就是合法的"——这个假设在 4.6-fast 面前碎了一地。

真正的修复

第一反应是在 CopilotX 里加映射、加后缀剥离逻辑。写了，测了，能跑。

但转念一想——claude-opus-4.6-fast 本来就不是真实模型。它是 Claude Code 配置里随手起的名字。CopilotX 的映射表应该处理格式差异（dash vs dot），不应该替虚构的模型名兜底。

所以回退了 CopilotX 的 hack，改了 Claude Code 的配置：

{
  "ANTHROPIC_SMALL_FAST_MODEL": "claude-haiku-4.5"
}

claude-haiku-4.5 是真实存在的模型，而且确实更快更便宜——本来就该用它。

最终 CopilotX 只保留了 vision header 的修复（正经 bug），模型映射保持干净。

复盘：为什么「修好了」还是崩

回头看日志，两个 bug 的报错其实一直同时存在：

错误表现根因
裸 Bad Requestvision 请求无 headerAnthropic 路由遗漏
model_not_supported虚构模型名透传dot-check 假设过强

但第一次排查时，裸 Bad Request 太显眼了——vision header 的问题一抓一个准，修完测试也过了。因为我的测试用的是正确的模型名。

只有让真正的调用方（Claude Code）跑一遍，双请求模式才暴露出第二个 bug。

教训很简单：测试能证明代码能跑，但不能证明场景覆盖了。 用 curl 测单张图片 vs Claude Code 跑完整对话流，覆盖面差了一个量级。

还有个彩蛋

排查过程中发现，CopilotX 的 logger.info() 日志全部被吞了——uvicorn 设了 log_level="info"，但那只影响 uvicorn 自己的日志。Python 的 logging.getLogger(__name__) 默认级别是 WARNING，所以 logger.info("Anthropic request: vision=%s ...") 从来没打印过。

这意味着我加的 vision debug 日志根本没输出。改成 logger.warning() 才看到。这是个小坑，但在紧急调试的时候，看不到日志会让人多绕好几分钟。

每次以为"这下该稳了"，下一个 400 就在路上了。不过说真的——裸 Bad Request 藏了两个独立 bug 这件事，比任何一个 bug 本身都有意思。

有时候修好了第一个 bug，不是结束，只是让第二个有了出场的机会。
