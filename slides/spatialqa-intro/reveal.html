必须讲清

两类数据，两套数据源

🏙 规划类（Planning）

  • 数据源：PostGIS
  • 规模：~120 图层（生产）
  • 对象：建筑、地块、规划边界、overlay
  • 场景：合规检查、开发完成度、空间分布

  • 数据源：SQLServer + ArcGIS SDE（生产）

Part 5

Agent 决策机制
混合策略为什么必要

两条查询路径

路径 A — MCP 工具

  • 适合：高频、确定性、可复用
  • 稳定 · 可控 · 可优化
  • 示例：query_buildings

路径 B — Text-to-SQL

  • 适合：长尾问题兜底
  • 灵活 · 覆盖未知问题
  • 示例：临时聚合/跨表组合
只用工具：覆盖率不够  |  只用 Text-to-SQL：稳定性不够
混合 = 主干稳定 + 尾部覆盖

写新能力时的判断框架

每次新增场景，先问自己这两个问题

这是高频主干场景吗？

→ 是：做成 MCP 工具，参数化、可测试、可复用

是偶发长尾场景吗？

→ 是：走 Text-to-SQL 兜底，不要过早工具化

工具或 SQL 报错了？

→ 错误信息回传模型修正，最多重试 3 次，超出后明确失败，不要悄悄错

Part 6

Schema 策略
与 SRID 性能大坑

Schema 注入策略

Demo

  • 全量注入
  • ~1.5K tokens，可控

生产

  • 核心 30 表常驻
  • 其余按主题动态注入
  • 规划 / 宜居 / 交通分批加载
三条硬规则
1. 注入不是越多越好
2. 目标是"足够决策"，不是百科全书
3. Schema 描述要写：业务含义 + 关键字段 + 常用筛选条件

SRID 与性能：我们踩过的大坑

12min+

跨 SRID 查询挂起时间

35s

SRID 统一 + 索引重建后

正确做法

  • 在数据层一次性统一 SRID（统一到 4326）
  • 重建 GIST 空间索引
  • 避免在高频 SQL 里临时 ST_Transform
性能公式：SRID 一致 + GIST 索引 + 查询可走索引 — 三者缺一不可

当前 SRID 状态

数据集SRID说明
规划类（pg_dump） 4326 (WGS84) 原始数据即 4326
宜居类（FileGDB） 32640 (UTM 40N) FileGDB 导入保留原 SRID
bus_stop（特殊） 4326 ← 原 32640 ALTER TABLE ST_Transform 永久转换
⚠️ Phase 2 接入 SQLServer 宜居类时，SRID 不一致是数据本身属性，不是导入问题。
需在 ETL 层一次性转换，而非运行时 ST_Transform。

Part 7

动手改造
标准流程与验收

新增工具场景的 5 步流程

  1. 定义问题与输出：明确输出字段、是否要图表、是否要 3D 高亮
  2. 写工具：spatialqa/tools/ 下新增模块，参数化 SQL（禁止字符串拼接）
  3. 注册与可发现：工具能被 Agent 发现，Schema 补齐触发描述
  4. 补测试：工具级（边界/空结果/非法参数）+ chat API 集成（自然语言触发）
  5. 按验收指标自检：首次命中率 · 平均耗时 · 输出完整性 · 错误可解释性

常见问题排查清单

现象先检查
模型选错工具Schema 描述是否模糊 · 工具命名是否打架 · 触发词是否缺失
查询变慢SRID 是否一致 · GIST/功能索引是否存在 · where 是否破坏索引
多轮答非所问session 截断是否过早 · 上轮摘要是否丢关键字段
图表不对图表类型 vs 数据类型 · category/value 映射是否反 · 空值处理
