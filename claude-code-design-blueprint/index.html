[CC解构] 设计蓝图：如果我们自己造一个

📅 2026-04-11⏱ 10 min read📝 1818 words

接上回：隐藏彩蛋：Buddy 系统、Undercover Mode、Feature Flags。

十一篇写完了。从一个 .map 文件的蝴蝶效应开始，到六层架构、七层系统提示词、40 个工具的统一抽象、两阶段安全分类器、三种并行模式、AsyncGenerator 驱动的核心循环、四类记忆分类法、结构化压缩、Skill/MCP 插件系统、React 终端 UI、Buddy 电子宠物。

现在回答最初的问题：如果要自己造一个 coding agent，怎么造？

架构蓝图

从 Claude Code 学到的第一课：分层，每层有明确的职责边界。

┌─────────────────────────────────────────┐
│ Surface: 终端 / IDE / Web / API         │
├─────────────────────────────────────────┤
│ Command: 用户意图解析 + Skill 调度      │
├─────────────────────────────────────────┤
│ Orchestration: 单 Agent / Fork / 编排   │
├─────────────────────────────────────────┤
│ Tool: 统一接口 + 安全分类 + 权限        │
├─────────────────────────────────────────┤
│ Engine: 主循环 + 流式 + 上下文管理      │
├─────────────────────────────────────────┤
│ Foundation: API 客户端 + 认证 + 沙箱    │
└─────────────────────────────────────────┘

数据向下流，结果向上回。每一层只跟相邻层通信。

十条核心原则

1. 系统提示词是产品设计，不是技术细节

系统提示词决定了 agent 的行为边界、代码哲学、安全准则、沟通风格。它应该被像产品需求文档一样认真对待——分层组织、有版本控制、有 owner、有 review 流程。

实施：把系统提示词分为静态（可全局缓存）和动态（session 级）两部分。安全相关的部分由专人维护。

2. 安全是架构的一部分，不是功能的一部分

每个 Tool 接口从诞生之日就包含 isReadOnly、toAutoClassifierInput、checkPermissions。不是先做功能再补安全——而是没有安全属性的工具无法被注册。

实施：把安全方法定义在 Tool 基础接口中，设为必需而非可选。

3. LLM 调用的经济学决定架构

Prompt cache 是一等公民。Agent 列表外移（省 10.2% cache creation token）、Fork 共享父 prompt cache、系统提示静态/动态分离——每一个设计决策都在考虑"这会花多少钱"。

实施：在架构评审中加一项"cache 影响分析"。追踪每个功能变更对 cache hit rate 的影响。

4. 用 LLM 审查 LLM

Auto Mode 的 yoloClassifier 证明了这个模式可行：用一个 LLM 实例来判断另一个 LLM 实例的操作是否安全。关键是两阶段（快速放行 + 深度复核）和不信任 agent 自己的文本输出。

实施：分类器只看客观操作记录（tool_use）和用户意图（user text），不看 agent 文本。安全默认：不确定就拒绝。

5. 并行是超能力，但需要纪律

Fork 几乎零成本启动，Coordinator 模式能同时指挥多个 Worker，但并行带来的复杂性需要严格的规则：防递归（fork 检测 boilerplate 标签）、并发安全标记（每个工具声明自己是否 safe）、错误传播（sibling abort）。

实施：每个 Agent 和 Tool 必须声明自己的并发属性。编排层根据声明做调度。

6. 流式一切

从 API 响应到工具执行到 UI 渲染——全链路流式。AsyncGenerator 是实现这一点的核心。用户感知的延迟是"第一个 token 到达的时间"，不是"整个响应完成的时间"。

实施：核心循环用 AsyncGenerator，工具执行器支持流式输入（模型还在输出时工具就开始执行）。

7. 压缩是有结构的遗忘

上下文窗口是有限的。五层上下文管理（Budget → Snip → Microcompact → Collapse → Auto Compact），每层过滤一些。9 段结构化模板确保压缩不丢关键信息。

实施：压缩不是"请总结"——而是一个结构化模板，指定保留什么、丢弃什么、以什么格式。

8. 记忆需要分类，不是堆积

四类记忆（user/feedback/project/reference）+ 索引/内容分离 + KAIROS 日志模式。没有分类的记忆会变成垃圾场；有分类的记忆是可检索、可过期、可共享的。

实施：定义记忆分类法。索引常驻上下文（200 行上限），内容按需加载。

9. 插件系统 = Prompt 的包管理器

Skill 的核心是一段 prompt。MCP 的核心是一个工具接口。两者共享同一套权限框架——外部工具零特权。

实施：Skill = prompt + 元数据 + 工具约束。MCP 工具走完整的安全流程。

10. 有趣的东西让产品有灵魂

Buddy 电子宠物、巫师注释、yolo 分类器的名字——这些不影响功能，但影响文化。一个让工程师乐在其中的产品，自然会是一个好产品。

最小可行架构

如果今天开始造，最小可行版本是什么？

Phase 1: 核心循环
  - 主循环（AsyncGenerator）
  - 3 个工具：FileRead, FileWrite, Bash
  - 基础权限（allow/deny）
  - 系统提示词（分静态/动态）

Phase 2: 安全和质量
  - Auto Mode（LLM 分类器）
  - Auto Compact（全量压缩）
  - CLAUDE.md 支持

Phase 3: 并行
  - Fork 子 Agent
  - 流式工具执行

Phase 4: 生态
  - MCP Client
  - Skill 系统
  - Memory 系统

Phase 5: Surface
  - IDE Bridge
  - 多 Surface 支持

Phase 1 可以用 500 行代码搞定一个原型。Phase 2 开始有生产可用性。Phase 3 开始有竞争力。Phase 4-5 开始有生态。

写在最后

12 篇博文，380,000 行源码，一个晚上。

写这个系列的过程本身就验证了一个信念：解构一个好产品是最快的学习方式。不是复制——而是理解每一个 why，然后在自己的场景里做出更好的决策。

Claude Code 不是完美的。它有 380K 行代码的复杂性，有 CACHED_MAY_BE_STALE 这样的技术债务暗示，有些 feature flag 后面的功能可能永远不会发布。

但它展现了一件事：认真做工程是什么样子的。

系统提示词不是一段话——是一本行为手册。安全不是一个 checklist——是架构的一部分。缓存不是优化——是商业模式。工具接口不是 (input) => output——是一个包含安全、权限、并发属性的完整契约。

这些不是 Claude Code 独有的智慧。它们是软件工程的智慧——只不过在 AI Agent 的语境下重新表达了一遍。

感谢那个忘记删 .map 文件的人。

朝圣之旅结束了。"耶路撒冷"不再神秘——但站在这里回望，它依然让人敬畏。
