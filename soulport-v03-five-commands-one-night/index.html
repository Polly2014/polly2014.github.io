接上回：SoulPort 完成了定位翻转——".bm 是标准格式，CLI 只是第一个消费者"。CLAUDE.md 被小龙虾评到 10/10。

文档到位了，代码该跟上了。

战场概览

今晚的目标：把 Roadmap 上 v0.2.x 和 v0.3 的所有命令实现出来。分工很简单——我写代码，小龙虾和 Baoli 做 Review。

小龙虾 Review 抓到第一个 bug：has_emoji 变量算了但没用。死代码。修完后五档等级阈值还跟 SoulArena 对齐了（🥚🐟🐋🦅🌊），底部加了 soul.polly.wang 引流。

第二刀：diff（灵魂差异）

absorb 之前需要知道这个包会改什么。soulport diff 对比 .bm 包和当前 workspace：

🔍 Soul Diff
Package  test.bm
Summary  +0 added · ~1 modified · ◦9 ws-only (kept) · =36 unchanged

⚙️ Config
  ~ TOOLS.md (1.7 KB → 1.7 KB)

📁 Projects
  ◦ BOOTSTRAP.md (1.4 KB, won't be changed)
  ◦ server.py.bak (25.1 KB, won't be changed)

小龙虾这轮最关键的 Review：原来"文件在 workspace 有但包里没有"的状态叫 removed——但 absorb 不会删这些文件！用户看到红色的 - 会以为东西要被删掉。改成 ws_only，符号用 ◦，提示 "won't be changed"。

命名不是小事，它决定了用户的心理预期。

第三刀：watch（灵魂守护）

这是定位分析里认定的"从工具到基础设施的质变"——让 SoulPort 从"换电脑时用一次"变成"每天都在跑"。

soulport watch --interval 6h --keep 10
  • 每 6 小时自动快照，保留最近 10 个
  • --once 模式：拍一张就走，适合 cron
  • skip-if-unchanged：workspace 没变就跳过，不浪费磁盘
  • 信号处理：Ctrl+C 优雅退出（sleep 拆 1 秒增量，不会卡 6 小时）

最重要的是 Soul Lineage——每个快照在 manifest.json 里记录 parent_hash：

snapshot-0033.bm: hash=d7fcf876..., parent=(root)
snapshot-0034.bm: hash=5a0ca89d..., parent=d7fcf876...

灵魂有了族谱。

小龙虾在这轮抓到了最严重的 bug：watch_loop 里用 _compute_file_hash(bm_file) 算 parent_hash，但那算的是 tar.gz 整体文件的 hash——而 get_parent_hash() 读的是 manifest 里的 content_hash。两个是不同的 hash，lineage 链会断！

修复：改用 _read_manifest(result).content_hash。一行代码，但没有 Review 就会一直默默错下去。

第四刀：changelog + rollback

有了快照，需要看变化、需要回滚。两个命令一起做。

changelog 对比连续快照生成变更摘要：

📜 Soul Changelog — last 1 change(s)

#1 1be975f7→e6f13bc3 (2026-03-26T16:57:02)
  ⚙️ config: ~1
    ~ TOOLS.md

rollback 通过 hash 前缀定位快照并恢复：

soulport rollback 1be975 --dry-run    # 先预览
soulport rollback 1be975 --force      # 再执行

小龙虾的 Review 又抓到两个设计缺陷：

  1. rollback 前没有备份。rollback 错了就回不去了——这是最大的安全风险。修复：自动 take_snapshot 一个 pre-rollback 备份（--no-backup 显式跳过）
  2. diff_packages 里文件消失用了 ws_only 状态。但两个包对比没有"workspace"的概念，文件确实消失了，应该用 removed。这是 diff 那轮改名的遗留问题——在一个语境下正确的命名，到另一个语境就错了

四轮 Review 的 4 个真 bug

轮次Bug严重度
doctorhas_emoji 死变量，算了不用P0
diffremoved 语义误导用户以为文件会被删P0
watch_compute_file_hash ≠ content_hash，lineage 链断裂P0-CRITICAL

每个 bug 都是"代码能跑，测试能过，但逻辑是错的"类型。这恰好是 Code Review 最大的价值——机器测试能抓到 crash，人（或 AI）Review 能抓到"不对劲"。
